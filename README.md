# peer — P2P screen + chat через Yandex Cloud

Кроссплатформенный (Windows / macOS) консольный клиент для обмена скриншотами и
сообщениями между двумя людьми за NAT/файрволами. В качестве relay используется
**Yandex Cloud Function** (без публичного IP-доступа), а в качестве хранилищ —
**YDB Serverless** (очередь сообщений и presence) и **Object Storage** (бинарные
JPEG скриншоты).

Архитектура подробно описана в Этапах 1–3 проектирования
(математическая модель → алгоритмы → паттерны). Здесь — только эксплуатация.

---

## Что вы получите

- `A` и `B` запускают `peer.py`, видят друг друга в логах как «онлайн».
- `A` жмёт **9+1** на клавиатуре (хоткей настраивается) → делается JPEG
  всего рабочего стола → грузится в Object Storage → `B` получает в консоли
  кликабельную ссылку «🖼 Новый скриншот».
- `B` пишет текст в консоль и нажимает Enter → у `A` появляется **системное
  уведомление** ОС с текстом.

---

## Требования

- **Python 3.10+** на обеих машинах.
- Аккаунт в [Yandex Cloud](https://console.yandex.cloud).
- На macOS — возможность дать терминалу разрешение **Accessibility** (нужно
  один раз).

## Часть 1. Развёртывание серверной стороны в Yandex Cloud

> Здесь всё делается один раз, любым из вас. Затем оба клиента используют
> один URL функции и один общий `API_KEY`.

### 1.1. Создайте облако и каталог

В консоли YC → создайте каталог, например `peer`. Дальше всё внутри него.

### 1.2. Создайте сервисный аккаунт

Сервисный аккаунт `peer-fn-sa` с ролями (минимально достаточный набор):

- `ydb.editor` — для работы с YDB
- `storage.editor` — для генерации presigned URL и доступа к Object Storage
- `serverless.functions.invoker` — если планируете вызывать из других мест

### 1.3. Создайте YDB Serverless

Консоль → **Managed Service for YDB** → создать БД **Serverless**.
Запишите:

- `YDB_ENDPOINT` — например `grpcs://ydb.serverless.yandexcloud.net:2135`
- `YDB_DATABASE` — путь вида `/ru-central1/.../{database-id}`

### 1.4. Создайте таблицы в YDB

Откройте Web SQL Console для созданной БД и выполните:

```sql
CREATE TABLE presence (
    nick      Utf8,
    last_seen Timestamp,
    PRIMARY KEY (nick)
);

CREATE TABLE messages (
    id         Utf8,
    src        Utf8,
    dst        Utf8,
    mtype      Utf8,
    payload    Json,
    created_at Timestamp,
    PRIMARY KEY (id),
    INDEX dst_created_idx GLOBAL ON (dst, created_at)
);
```

Опционально (рекомендуется) — настройте TTL для очистки старых записей
через UI YDB или через `ALTER TABLE messages SET (TTL = ...)`.

### 1.5. Создайте Object Storage bucket

Консоль → **Object Storage** → создать бакет, например `peer-screens`.
Дополнительно:

- Доступ — **приватный**.
- (Рекомендуется) Lifecycle: правило «удалять объекты старше 1 дня».

### 1.6. Создайте статический ключ для S3-доступа

Сервисные аккаунты → ваш `peer-fn-sa` → **Создать статический ключ**.
Запишите `AWS_ACCESS_KEY_ID` и `AWS_SECRET_ACCESS_KEY`.

### 1.7. Создайте Cloud Function

Консоль → **Cloud Functions** → создать функцию `peer-relay`.
Затем создаём **версию**:

- **Среда выполнения:** Python 3.12 (или 3.11)
- **Точка входа:** `handler.handler`
- **Таймаут:** 10–30 сек
- **Память:** 256 МБ хватит (можно поднять до 512 при росте трафика)
- **Сервисный аккаунт:** `peer-fn-sa` (созданный выше)
- **Переменные окружения:**

  ```
  API_KEY=<придумайте_длинный_общий_секрет>
  YDB_ENDPOINT=grpcs://ydb.serverless.yandexcloud.net:2135
  YDB_DATABASE=/ru-central1/.../...
  S3_BUCKET=peer-screens
  S3_ENDPOINT=https://storage.yandexcloud.net
  S3_REGION=ru-central1
  AWS_ACCESS_KEY_ID=...
  AWS_SECRET_ACCESS_KEY=...
  PEER_TTL_SEC=15
  PRESIGN_TTL_SEC=900
  ```

- **Файлы:** залейте `handler.py` и `requirements.txt` (см. ниже).

### 1.8. Содержимое деплоя

В архив (zip) положите два файла:

**handler.py** — серверная функция (есть в репозитории).

**requirements.txt:**

```
ydb>=3.0
boto3>=1.34
```

Залейте архив через «Создать версию → загрузить ZIP» или через `yc serverless`.

### 1.9. Сделайте функцию публично вызываемой

Тип триггера — **HTTP**. В разделе функции → **Сделать функцию публичной**
(или повесьте API Gateway, но для приватных нужд достаточно публичной функции
с защитой через наш `API_KEY` в теле запроса).

Запишите выданный URL вида:

```
https://functions.yandexcloud.net/d4e...
```

Это ваш `RELAY_URL`.

### 1.10. Быстрая проверка

```bash
curl -X POST <RELAY_URL> \
  -H "Content-Type: application/json" \
  -d '{"op":"heartbeat","api_key":"<API_KEY>","user":"test"}'
```

Должно вернуться:

```json
{ "ok": true, "peers": [] }
```

Если `ok: false` — посмотрите логи функции в консоли YC, там будет точная причина.

## Часть 2. Установка клиента

На каждой машине (Windows и macOS):

### 2.1. Установите Python зависимости

```bash
python -m pip install --upgrade pip
python -m pip install -r requirements-client.txt
```

### 2.2. macOS: выдайте Accessibility-разрешение

Для перехвата глобального хоткея `pynput` нужно, чтобы **терминал** имел
доступ к Accessibility:

1. **System Settings** → **Privacy & Security** → **Accessibility**.
2. Нажмите `+` и добавьте ваш терминал (Terminal.app / iTerm2 / WezTerm / ...).
3. Тумблер должен быть **включён**.
4. Перезапустите терминал.

Если этого не сделать — клиент запустится, но хоткей `9+1` не сработает,
в логах увидите соответствующее сообщение.

### 2.3. Запуск

Самый простой вариант — через переменные окружения:

**macOS / Linux:**

```bash
export RELAY_URL='https://functions.yandexcloud.net/...'
export API_KEY='ваш_общий_секрет'
export NICK='alice'
export HOTKEY='9+1'        # опционально, по умолчанию 9+1
export LOG_LEVEL=INFO      # опционально: DEBUG/INFO/WARNING
python peer.py
```

**Windows (PowerShell):**

```powershell
$env:RELAY_URL = 'https://functions.yandexcloud.net/...'
$env:API_KEY   = 'ваш_общий_секрет'
$env:NICK      = 'bob'
python peer.py
```

Можно также запустить без переменных — программа спросит интерактивно.

## Часть 3. Как пользоваться

После запуска вы увидите баннер и логи:

```
17:42:01.123 [INFO   ] main       │ Никнейм: 'alice'   Хоткей: 1+9   Relay: https://...
17:42:01.456 [INFO   ] platform   │ Определена ОС: Darwin 23.5.0
17:42:01.789 [INFO   ] heartbeat  │ Старт: τ=5.0s, Θ=15.0s (TTL для других)
17:42:02.012 [INFO   ] poller     │ Старт: p=2.0s
17:42:02.345 [INFO   ] hotkey     │ Слушаю глобальный хоткей: 1+9
17:42:06.678 [INFO   ] heartbeat  │ Онлайн появились: bob
```

### Команды в консоли

| Команда       | Что делает                       |
| ------------- | -------------------------------- |
| `/peers`      | Показать список онлайн-пиров     |
| `/to <nick>`  | Выбрать собеседника              |
| `/quit`       | Завершить программу              |
| `любой текст` | Отправить выбранному собеседнику |

Если онлайн ровно один человек — он выбирается автоматически.

### Скриншот

В любой момент нажмите **9+1** одновременно. В логах увидите:

```
17:43:21.000 [INFO   ] hotkey     │ Хоткей 1+9 сработал
17:43:21.100 [INFO   ] coord      │ HOTKEY → CAPTURE
17:43:21.250 [INFO   ] capture    │ Снимок сделан: 3840x2160, 412.3 KB, JPEG q=85
17:43:21.300 [INFO   ] coord      │ CAPTURE_OK → UPLOAD к 'bob'
17:43:21.400 [INFO   ] upload     │ Запрашиваю presigned URL на upload, key=...
17:43:21.700 [INFO   ] blob       │ Загружено в Object Storage: 412.3 KB
17:43:21.900 [INFO   ] upload     │ Уведомляю 'bob' о новом скриншоте
```

У собеседника появится в консоли:

```
🖼  Новый скриншот от alice
   URL: https://storage.yandexcloud.net/...
```

Текст со значком — кликабельная ссылка (Ctrl/Cmd+клик в большинстве терминалов).
Открывается в браузере, изображение сразу видно.

## Часть 4. Диагностика проблем

Логи структурированы по компонентам — это позволяет быстро находить, **где
именно** что отвалилось. Слева в логе всегда видно имя подсистемы.

| Компонент    | За что отвечает                           |
| ------------ | ----------------------------------------- |
| `main`       | Старт, инициализация, выключение          |
| `platform`   | Определение ОС, создание Capture/Notifier |
| `heartbeat`  | Отметка «я живой» + получение peers       |
| `poller`     | Получение входящих сообщений              |
| `hotkey`     | Перехват клавиатуры                       |
| `console`    | Чтение stdin                              |
| `coord`      | FSM: переходы состояний                   |
| `capture`    | Скриншот                                  |
| `upload`     | presigned URL + PUT + send notification   |
| `blob`       | Сетевой PUT в Object Storage              |
| `screenshot` | Получение download URL и вывод ссылки     |
| `notify`     | Системные уведомления                     |
| `text`       | Отправка TEXT-сообщений                   |
| `retry`      | Любые повторные попытки сетевых вызовов   |
| `relay`      | HTTP-запросы к Cloud Function             |

### Типовые проблемы

**1. «HTTP 401 / unauthorized»** в логах `retry`/`relay`
→ Несовпадение `API_KEY` у клиента и в переменных окружения функции.

**2. «HTTP 5xx server error» с retriable=true**
→ Скорее всего YDB или S3 вернули ошибку. Откройте логи функции в консоли YC.
Частые причины:

- Сервисный аккаунт не имеет нужной роли.
- Таблицы в YDB не созданы.
- Имя бакета или endpoint S3 неверны.

**3. Хоткей не срабатывает на macOS**
→ Terminal не имеет Accessibility-разрешения. См. §2.2.

**4. Хоткей не срабатывает на Windows**
→ Запустите терминал **от имени администратора**. Некоторые игры/программы
с глобальным захватом ввода тоже могут перехватывать клавиатуру.

**5. Скриншот «пустой» / чёрный на macOS**
→ Терминалу нужно ещё одно разрешение — **Screen Recording**.
System Settings → Privacy & Security → Screen Recording → добавить терминал.

**6. Heartbeat проходит, peers пустые, хотя оба запущены**
→ Подождите ~5 секунд (один цикл heartbeat).
→ Проверьте, что в YDB-таблице `presence` обе записи появились (SQL Console).
→ Убедитесь, что `PEER_TTL_SEC` (по умолчанию 15) больше `τ=5`.

**7. Сообщения приходят, но дубликатами**
→ Не должно быть из-за `DedupCache` на клиенте. Если видите —
откройте issue, приложите логи на уровне `DEBUG`.

**8. Cold start: первый запрос долгий (~1-3 секунды)**
→ Это нормально для YCF. Дальнейшие запросы быстрые. При желании можно
настроить **provisioned instances** в функции, но это платно.

### Включить максимально подробные логи

```bash
LOG_LEVEL=DEBUG python peer.py
```

## Часть 5. Безопасность (что есть и чего нет)

- **TLS до relay** — да, всегда (`https://functions.yandexcloud.net`).
- **TLS до Object Storage** — да (presigned URLs указывают на https).
- **Shared API_KEY** — единственная защита от посторонних. Не публикуйте его.
- **E2E-шифрование** — нет. Сам бакет приватный, но содержимое JPEG проходит
  через relay-инфраструктуру YC в незашифрованном виде. Для друзей этого
  достаточно; если нужно больше — обвязать `cryptography` (AES-GCM с общим
  ключом) над bytes до upload и расшифровка после download — относительно
  просто, могу добавить отдельно.
- **Аутентификация пользователей по никнейму** — отсутствует: кто угодно
  с `API_KEY` может представиться любым ником. Опять же — для друзей это
  ок, для продакшена — нужно вешать JWT или другую схему.

## Часть 6. Стоимость в Yandex Cloud (порядок)

При интенсивности «несколько скриншотов в час между двумя людьми» — расходы
**близки к нулю**:

- YCF: 1 млн бесплатных вызовов/месяц. Heartbeat (12/мин × 2 человека × 30 дней)
  ≈ 1 млн в месяц — уложитесь даже на бесплатном тарифе, если активны не 24/7.
- YDB Serverless: первые единицы RU и ГБ — бесплатно.
- Object Storage: до 1 ГБ free, дальше копейки.

Реальные цены смотрите на [актуальной странице тарифов YC](https://yandex.cloud/ru/prices).
